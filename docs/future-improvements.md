# 160Grab 后续最值得做的改进

这份文档的目标不是列一个“大而全”的功能愿望清单，而是基于三个现实前提，给出更值得落地的后续方向：

- 参考项目 `91160-cli` 已经证明了哪些能力在长期使用中有价值
- 当前 `160Grab` 已明确选择了“手动接管 + 真实浏览器 + 浏览器上下文内请求 + 页面预约”路线
- `docs/superpowers/handoffs/2026-03-23-91160-manual-takeover.md` 已经把现阶段的已知风险和未完成项写得很清楚

因此，后续改进的判断标准不是“缺什么补什么”，而是：

- 优先补当前主链路最脆弱的部分
- 优先吸收 `91160-cli` 中和当前架构兼容的成熟思路
- 不为了功能面而破坏现在这条更贴近真实浏览器的路线

## 推荐优先级

### 1. 建立真实站点 smoke 基线，并定期刷新夹具

这是当前最值得做的第一件事。

原因很直接：现有 handoff 已经明确写出 4 个高风险点，分别是医生详情页 URL、`member.html` DOM、`/guahao/v1/pc/sch/doctor` 返回结构，以及预约页隐藏字段/成功判定。如果这几个地方没有真实站点基线，后续无论补第二通道、通知、甚至自动登录，都会建立在可能已经过时的假设上。

这项改进应该重点落在：

- 用一个稳定的 smoke 流程捕获真实医生页、`member.html`、排班返回和预约页 HTML
- 把真实样本回灌到 `tests/fixtures/`
- 让 `tests/services/test_session.py`、`tests/services/test_schedule.py`、`tests/services/test_booking.py` 更贴近真实页面结构
- 把“什么时候需要刷新夹具”写进文档，而不是只靠临时排查

涉及文件：

- `src/grab/services/session.py`
- `src/grab/services/schedule.py`
- `src/grab/services/booking.py`
- `tests/fixtures/`
- `tests/e2e/`

### 2. 在持久化 profile 路线上补 branded browser 支持

当前项目已经做对了一半：默认使用 `launch_persistent_context`，并把 profile 持久化作为主链路。

下一步最有价值的不是先上代理池，而是把浏览器本体再向真实用户环境靠近一步，例如支持：

- `browser.channel: chrome`
- `browser.channel: msedge`
- 可选的启动参数、语言、时区或窗口尺寸配置

这项改进的价值在于：

- 它和当前“人工操作结合脚本”的路线一致
- 它比“先做 CDP attach”更符合现有 `docs/compare_with_attach.md` 的结论
- 它比“先做代理”更不容易把问题从页面一致性转移到运维复杂度

涉及文件：

- `src/grab/models/schemas.py`
- `config/example.yaml`
- `src/grab/browser/playwright_client.py`
- `README.md`

### 3. 做一个适配手动接管模型的轻量 init / 写回助手

`91160-cli` 的 `init` 之所以有价值，不在于它必须生成 `config.properties`，而在于它降低了用户第一次使用的配置成本。

`160Grab` 现在已经具备一部分基础能力：

- 能自动解析当前医生页 URL
- 能自动选择或校验 `member_id`
- 已经支持把 `browser.profile_name` 写回配置

所以更值得做的不是完整复刻 `91160-cli init`，而是补一个更轻量、贴合当前路线的写回助手，例如：

- 首次成功选择 profile 后，询问是否写回 `browser.profile_name`
- 首次成功选择就诊人后，询问是否写回 `member_id`
- 在用户已经手动打开目标医生页的前提下，可选写回最近一次成功使用的目标信息或过滤条件

这类改进能明显降低重复操作，但不会把项目重新拉回“先初始化一堆城市/医院/科室 ID”的老路。

涉及文件：

- `src/grab/utils/config_writer.py`
- `src/grab/services/session.py`
- `main.py`
- `README.md`

### 4. 增加成功通知、结构化日志和更好的运行可观测性

`91160-cli` 里真正值得借鉴的，不只是代理或双通道，还有它作为“长时间运行工具”的使用体验，例如通知和日志。

对 `160Grab` 来说，最值得补的是：

- 预约成功后的本地通知或 webhook 通知
- 更结构化的日志输出，方便区分“登录问题 / 页面结构变化 / 风控限频 / 预约提交失败”
- 对长期刷号过程给出更清晰的状态摘要，而不是只靠滚动日志

这项工作对当前路线非常友好，因为它不改变主流程，只提升可用性和排障效率。

如果只做一个最小版本，建议先从下面两类通知开始：

- 本地桌面通知
- 简单 webhook 或 Server酱 / PushDeer 一类单向通知

### 5. 把第二刷号通道作为 fallback，而不是直接恢复成默认双通道

`91160-cli` 支持通道 1 / 通道 2，不是历史包袱，而是因为站点不同页面的放号信息确实可能不完全一致。

所以“恢复第二通道”本身是有价值的，但不应该马上照搬成默认双通道主路径。对 `160Grab` 来说，更合理的做法是：

- 继续把当前医生详情页通道作为主路径
- 只有在当前接口长期空返回、结构异常或明显漏号时，才启用 fallback
- 保持请求仍然走浏览器上下文，而不是退回浏览器外长连接刷接口

也就是说，值得借鉴的是 `91160-cli` 的“多通道兜底思想”，不是它的 HTTP-first 实现方式。

涉及文件：

- `src/grab/services/schedule.py`
- `src/grab/models/schemas.py`
- `tests/services/test_schedule.py`
- `README.md`

## 目前不建议优先做的事情

下面这些并不是永远不做，而是当前阶段不值得排到前面。

### 1. 把 `auto` 登录做成主路径

虽然 `91160-cli` 已经有 OCR 登录链路，当前仓库里也保留了 `ocr.py` 和 `auth.strategy = auto` 的入口，但这件事现在的收益并不高。

原因是：

- 它和当前“人工接管”路线的核心价值并不一致
- 维护成本高，且最容易被站点页面/验证码策略变化击穿
- 如果真实医生页、成员页、预约页都还没建立稳定基线，先补自动登录只会扩大不稳定面

更合理的定位是：**保留为 fallback，而不是短期主攻方向。**

### 2. 先上代理池

`91160-cli` 的代理能力很实用，但对当前项目来说，先做代理并不是最高 ROI。

在 `160Grab` 的路线里，更优先的问题是：

- 浏览器环境是否足够接近真实用户
- 页面请求和页面提交流程是否稳定
- 节流和退避是否足够合理

在这些基础问题还没完全打稳之前，先引入代理池，往往只是把问题从“页面行为”换成了“网络和运维行为”。

### 3. Docker-first 运行形态

`91160-cli` 的 Docker 包装很合理，因为它是 CLI + HTTP 请求主导。

但 `160Grab` 当前的核心是：

- headed browser
- 持久化 profile
- 用户手动接管登录和医生页导航

这套模型天然更贴近本机桌面环境，而不是容器优先环境。除非后面明确要做远程托管和图形会话管理，否则 Docker 不应该排在前几位。

## 推荐推进顺序

如果只看“下一阶段最值得做什么”，建议顺序如下：

1. 真实站点 smoke 和夹具刷新
2. branded browser 支持
3. 轻量 init / 写回助手
4. 通知与可观测性
5. 第二通道 fallback
6. `auto` 登录 fallback
7. 代理 / Docker 等更重的运行时能力

这个顺序的核心逻辑是：

- 先把当前主链路验证稳
- 再提升它的真实性和易用性
- 最后才补更重、更宽、维护成本更高的能力
