# 为什么推荐 160Grab

如果你正在找一个更贴近真实使用方式的健康160挂号脚本，`160Grab` 值得试一下。

它不是把人完全踢出流程的“纯自动化黑盒”，而是把最像真人、最容易触发风控的步骤留给你自己完成，再把最重复、最耗时的刷号和预约流程交给脚本。这种“人工接管 + 浏览器自动化”的组合，正是它和传统 CLI 挂号脚本最大的不同。

## 它解决的是什么问题

很多脚本在两个方向上都不够理想：

- 要么太“纯接口”，配置复杂、容易和真实页面状态脱节
- 要么太“全自动”，登录、验证码、跳页、提交全部硬控，实战里更容易脆

`160Grab` 选择了第三条路：

- 用真实 Chromium 浏览器承载整个会话
- 由你手动登录账号、手动进入目标医生详情页
- 由脚本接管后续的轮询、筛选、打开预约页、回填表单、提交重试

这个模式非常适合“我愿意自己完成关键操作，但不想手工一直刷新页面”的用户。

## 这个项目有什么吸引力

### 1. 更贴近真人操作

项目默认使用 Playwright 驱动真实浏览器，并支持持久化 profile。你可以复用自己的登录状态、Cookie 和常用浏览环境，而不是每次都从零开始造一套脚本会话。

再配合 `playwright-stealth`、页面内请求、随机节流和命中“访问次数过多”后的冷却退避，整体策略比“浏览器外部死刷接口”更稳健。

### 2. 人工和脚本的分工很合理

你负责：

- 登录
- 确认账号
- 打开目标医生详情页

脚本负责：

- 读取当前医生页 URL
- 提取 `unit_id / dep_id / docid`
- 校验或选择就诊人
- 持续刷号
- 命中条件后打开预约页
- 选择可约时段并提交

这种分工的好处是：脚本不用去碰最脆弱的登录和页面导航，而你也不用再手工做高频重复刷新。

### 3. 上手门槛比想象中低

你不需要先跑一个复杂的初始化向导，也不需要先把城市、医院、科室、医生这些 ID 全部手动抄进配置。

在 `160Grab` 里，更自然的做法是：

1. 创建浏览器 profile
2. 启动脚本
3. 手动登录
4. 手动进入目标医生详情页
5. 回终端按 Enter，让脚本接管

对于实际用户来说，这往往比“先研究一堆编号和配置项”更顺手。

### 4. 真正适合排查问题

真实站点变动时，最麻烦的不是“脚本失败”本身，而是你不知道它失败在哪。

`160Grab` 支持通过 `GRAB_DEBUG_DIR` 落盘：

- 当前页面 HTML
- 全页截图
- 最近的 console / pageerror / requestfailed 事件

这让它不仅是一个“能跑”的脚本，也更像一个“能诊断”的工具。

## 适合谁

- 想保留人工登录，不想把账号密码长期写进配置的人
- 需要真实浏览器参与，不信任纯 HTTP 挂号脚本的人
- 愿意自己完成关键一步，但不想继续手工刷号的人
- 想基于 Python + Playwright 继续扩展自己流程的人

## 当前版本最值得注意的点

- 默认主链路已经切到“手动登录接管”
- 当前优先支持医生详情页通道
- 预约提交走真实页面策略
- `auto` 登录入口还在，但并不是当前主打路径

这不是缺点，而是项目当前的明确取舍：先把最常用、最贴近真实浏览器的主链路打磨稳，再考虑补更宽的能力面。

## 快速开始

```bash
uv sync
uv run playwright install chromium
cp config/example.yaml config.yaml
uv run python main.py config.yaml --create-profile
uv run python main.py config.yaml
```

如果你想要的是一个“人负责关键动作，脚本负责重复动作”的健康160挂号工具，`160Grab` 正好命中这个场景。
